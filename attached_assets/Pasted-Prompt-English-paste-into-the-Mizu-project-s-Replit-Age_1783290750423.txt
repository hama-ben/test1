This makes that same client — the one every Realtime subscription and direct query already uses — actually carry the user's real identity instead of running as anonymous. Do this in one central place (e.g. inside use-auth.ts's login/signup success handlers and its app-boot initialization effect), not scattered across every page that happens to query Supabase.
Step 4 — Keep the session in sync during token refresh
token-refresh.ts / use-token-refresh.ts already rotate the access token in the background via POST /api/auth/refresh. After a successful refresh, also call supabase.auth.setSession() again with the new tokens, so the Supabase client's internal session never goes stale relative to what the backend considers valid.
Step 5 — Handle logout correctly
On logout, call supabase.auth.signOut() on the frontend client (in addition to whatever the app already does to clear its own backend session/local storage), so the Supabase client drops back to anonymous cleanly rather than holding a dead session.
