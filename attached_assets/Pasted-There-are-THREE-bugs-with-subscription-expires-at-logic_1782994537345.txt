There are THREE bugs with subscription_expires_at logic:

═══════════════════════════════════════════════════════════════
## BUG 1 — Payment approval doesn't STACK on existing subscription
═══════════════════════════════════════════════════════════════

When admin approves a payment receipt, the driver should get 
+30 days ADDED ON TOP of their current subscription_expires_at 
(not from NOW()).

Current behavior: 
  Driver has 32 days remaining → admin approves payment → 
  result shows only 30 days (from NOW()) instead of 62 days 
  (32 existing + 30 new)

Required behavior:
  new_expiry = MAX(subscription_expires_at, NOW()) + 30 days
  
  This means: take whichever is later (current expiry OR now), 
  then ADD 30 days on top of it — always stacking, never resetting.

Fix the payment approval atomic SQL to:
  UPDATE users
  SET subscription_expires_at = 
    GREATEST(subscription_expires_at, NOW()) + INTERVAL '30 days'
  WHERE id = :driverId
  RETURNING subscription_expires_at;
