CONFIRMED BUG SOURCE — traced from the Al-Shaibia Admin project
═══════════════════════════════════════════════════════════════
Direct database evidence: rejecting a payment on the admin panel
causes subscription_expires_at for the driver to drop from a future
date (~60+ days out) down to approximately NOW() + 3 days —
immediately, even though the admin backend's reject endpoint
provably does NOT touch subscription_expires_at (confirmed by full
code review on the admin side).

The admin's rejection flow does two things only:
1. Sets subscription_payments.status = 'rejected'
2. Inserts a row into "announcements" with target_audience = driver_id

The leading theory, confirmed from the admin side, is that THIS
project (the driver app) has a Realtime listener — likely in
use-subscription-notifications.ts — that reacts to a payment
rejection event (either by listening on subscription_payments
status changes, or on the announcements insert) and then calls some
endpoint that writes subscription_expires_at = NOW() + 3 days
UNCONDITIONALLY, overwriting the driver's real remaining balance.

STEP 2 — FIX (after confirming)
═══════════════════════════════════════════════════════════════
Whatever endpoint is found writing the unconditional 3-day grace
period on rejection must be changed to the same safe pattern already
proven correct elsewhere in this project:

UPDATE users
SET subscription_expires_at = CASE
  WHEN subscription_expires_at IS NULL
    OR subscription_expires_at < NOW() + INTERVAL '3 days'
  THEN NOW() + INTERVAL '3 days'
  ELSE subscription_expires_at
END
WHERE id = :driverId;

This must NEVER reduce an existing future subscription_expires_at,
regardless of what triggered it.
